## 작성 가이드라인

### 1. User Story 작성
- **As a**: 구체적인 사용자 역할 명시
- **I want to**: 명확하고 측정 가능한 기능
- **So that**: 비즈니스 가치와 연결된 이유

### 2. Acceptance Criteria
- **구체적**: 모호하지 않은 명확한 조건
- **측정 가능**: 완료 여부를 객관적으로 판단 가능
- **테스트 가능**: 실제 테스트로 검증 가능

### 3. Test Cases
- **긍정적 테스트**: 정상 동작 시나리오
- **부정적 테스트**: 오류 상황 시나리오
- **경계값 테스트**: 극한 상황 시나리오

### 4. Story Point 추정
- **1점**: 매우 간단한 작업 (1-2시간)
- **2점**: 간단한 작업 (2-4시간)
- **3점**: 보통 작업 (4-8시간)
- **5점**: 복잡한 작업 (1-2일)
- **8점**: 매우 복잡한 작업 (2-3일)
- **13점**: 거대한 작업 (3-5일)

## 품질 기준

### 좋은 User Story의 특징
- **독립적**: 다른 Story와 독립적으로 구현 가능
- **협상 가능**: 개발팀과 협의하여 세부사항 조정 가능
- **가치 있는**: 사용자에게 실제 가치 제공
- **추정 가능**: Story Point로 크기 추정 가능
- **작은 크기**: 1-2 스프린트 내 완료 가능

### 피해야 할 것
- **너무 큰 크기**: 여러 스프린트에 걸치는 작업
- **모호한 표현**: "좋은", "빠른", "사용자 친화적" 등
- **기술적 세부사항**: 구현 방법보다는 무엇을 할지에 집중
- **의존성 과다**: 다른 Story에 과도하게 의존
